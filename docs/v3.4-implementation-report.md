# v3.4 Implementation Report

v3.4 turns the v3.3 scanner into a control-plane producer. The manager API now
owns asynchronous scan jobs, normalized assets, evidence, assessments and
migration plans in the same SQLite store used by releases and audit events.

The end-to-end workflow is:

```text
C++/RSA/OpenSSL discovery
  -> persistent CryptoAsset and evidence
  -> migration assessment
  -> compatibility Service/Policy
  -> immutable release and gateway-agent health
  -> verification/fallback gate
  -> strict X25519MLKEM768-only release
  -> VERIFIED migration state
```

Static discovery does not invent service ports or upstream ownership. The
migration request must supply those deployment facts, normally from CMDB. A
strict release cannot be staged until the compatibility release is `HEALTHY`,
the caller provides a successful verification result and the measured fallback
rate is below the plan threshold.

Advanced C++ evidence now combines compilation database metadata, direct source
interfaces, bounded macro expansion, heuristic wrapper-call propagation, ELF
and static archive symbols, C++ name demangling and optional runtime eBPF events.
Each signal retains its evidence source and confidence. Compilation database
commands and target programs are never executed.

Verification added in this release:

- 21-case enterprise scanner matrix;
- 10-case scan-to-migration REST API matrix;
- offline unit coverage for C++ compile context, macro/call graph, static
  archives, demangling, eBPF trace ingestion and strict promotion gates;
- all v3.1-v3.3 compatibility, release, API, scanner and gateway tests.

The release remains single-node. Distributed scan workers, PostgreSQL, multi-
gateway rollout coordination and privileged eBPF deployment policy remain
future work.
